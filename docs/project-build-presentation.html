◆ Phase 2 — Build

Build 1: The Chrome Extension

Claude wrote content.js and manifest.json. The extension handles everything inside the YouTube tab — nothing else.

What it does
Tracks pointer position across the YouTube page continuously. When hovering a thumbnail, finds the underlying video link — even when YouTube's overlay elements obscure the direct path. On trigger, copies the clean URL and shows a toast.
The tricky part Claude solved
YouTube uses custom web components and hover-preview overlays that sit in front of the actual link. The extension walks up the DOM and, if the direct path fails, searches the nearest video container as a fallback. Covers regular videos, Shorts, playlists, sidebar.
▶ First test at this stage
Load the extension in Brave, open YouTube, hover thumbnails, press the trigger key. Confirm the "Copied!" toast appears and the right URL is in the clipboard — before writing the AHK script at all.
💡 Build and test one component at a time
Don't wait until everything is written to test. Each component should be verified standalone first. This makes bugs much easier to locate.
◆ Phase 2 — Build

Build 2: The AutoHotkey Orchestrator

Claude wrote copy.ahk — the glue between YouTube, the extension, and Gemini.

1
Find the right windowsLocates the Brave window for YouTube and the Chrome window titled "Gemini." Uses process name + window class + title filters to avoid false matches.
2
Trigger the extensionActivates Brave, syncs hover state so the page registers the current thumbnail, then sends the trigger key to the extension.
3
Wait for the URL reliablyInstead of guessing with a timer, it watches the Windows clipboard sequence number — only reading the clipboard once the OS confirms a change happened.
4
Deliver to GeminiActivates the Gemini PWA window, clicks the composer, pastes the prompt + URL, hits Enter. Clipboard is then restored to the plain URL.
💡 Ask Claude to explain design choices
"Why watch the clipboard sequence number instead of using a timer?" confirms Claude thought through the timing problem — and teaches you something useful for future projects.
◆ Phase 2 — Build

Build 3: Wiring Up Gemini

Clicking inside a Chrome PWA without moving the mouse required a specific approach.

The challenge
Standard AHK click commands move the physical cursor. The goal was to land a click in Gemini's composer input while the user's mouse stays wherever it was — no visible disruption.
Claude's approach
Send raw Windows mouse messages directly to Chromium's render widget via PostMessage. The click lands in Gemini's UI; the OS cursor never moves.
⚡ Configurable by design
Claude added two named constants at the top of the script for the composer click position (as fractions of the window size). If the click lands slightly off, you change two numbers — no need to read the rest of the script.
💡 Ask Claude to make tunable values obvious
"Put anything someone might need to adjust as named constants at the top" is a prompt worth repeating for any project. It makes the result usable by non-developers.
◆ Phase 3 — Iterate

Iteration 1: From Three Hotkeys to One

The first version had three separate shortcuts. Through review and conversation, it became one.

⚠ First version — three hotkeys
Alt+X — copy URL only (extension trigger, global)
Alt+C — full flow to Gemini
Alt+Z — paste only (legacy from an older script)
💡 The problem
Three shortcuts with overlapping jobs added cognitive load. The paste-only Alt+Z was a leftover from an earlier version (paste.ahk) that no longer existed as a separate file.
✓ After iteration — one user hotkey
Alt+Z — the full flow: copy from YouTube → wait for clipboard → activate Gemini → click composer → paste prompt + URL → send. That's it.
⚠ Internal trigger later swapped to F24
AHK still needs to signal the extension inside the tab. Originally that was a synthetic Alt+X, but another global tool (CopyAnkitoChatGPT) owned Alt+X system-wide and was swallowing it. Switched the internal handshake to F24 — no default browser binding, no possible collision. Alt+Z stays the only key the user ever presses.
◆ Phase 3 — Iterate

Iteration 2: Stop Restoring the Cursor, Stop Moving It

The first attempt at "no visible cursor jump" was solving the wrong problem.

1
First attempt: save and restore the cursor position
AHK saved the mouse position before the Gemini click, then jumped back in a cleanup block. Technically worked — but the cursor visibly snapped back to where it started. Still jarring.
2
Realisation: wrong problem being solved
The real goal was "don't move the mouse at all" — not "move it and jump it back." Described this to Claude: "The cursor snapping back looks bad. I don't want it to move in the first place."
✓
Final: PostMessage — cursor never moves
Raw Windows messages sent directly to Chromium's render widget. No physical mouse movement at any point. The save/restore logic and all the mouse-position code was removed entirely.
💡 Describe the outcome, not the fix
"The cursor snaps back which looks bad" led Claude to the right solution. "Make the restore faster" would have produced a worse version of the wrong approach.

◆ Phase 4 — Debug

Bug 1: Gemini Sometimes Got the Wrong URL

Intermittent bugs are the hardest kind. This one happened about 1 in 5 times.

🟥 The symptom
After Alt+Z, Gemini occasionally received a URL from a previous press — not the current thumbnail. The "Copied!" toast showed in YouTube every time, so the extension appeared to be working fine.
The bug report given to Claude
"When I hover thumbnail A and press Alt+Z, Gemini sometimes gets a URL from a previous session. Happens maybe 1 in 5 times. The toast shows 'Copied!' every time so the extension side seems fine."
✓ Root cause and fix
The browser clipboard API is asynchronous — the write hadn't finished before AHK read the clipboard. Fix: switch to a synchronous execCommand("copy") path. AHK also moved from a guessed timer to watching the Windows clipboard sequence number — only proceeding once the OS confirms the write happened.
🟥 Months later: "sometimes works, sometimes doesn't" returned
Same symptom resurfaced intermittently. Built an instrumentation harness (ring-buffer in content.js + document.title beacon read by AHK + per-stage timing logs in copy.ahk, all behind flags). The beacon revealed the extension was never receiving the keydown at all on failed runs.
✓ Real root cause: hotkey collision
The internal trigger was synthetic Alt+X. A separate productivity tool (CopyAnkitoChatGPT) had a global Alt+X hook that intercepted the synthetic event before Brave saw it. Swapped the internal handshake to F24 — zero collisions possible. Reliability went from "flaky" to consistent.
💡 Always include frequency in bug reports
"Happens 1 in 5 times" immediately tells Claude this is a race condition or timing issue — not a logic error. That single detail narrowed the diagnosis significantly.
◆ Phase 4 — Debug

Bug 2: Hover Detection Failed on Some Thumbnails

YouTube uses different page structure depending on where you are on the site.

🟥 The symptom
On Shorts thumbnails on the home page and playlist items, Alt+Z did nothing — no toast appeared. Regular video thumbnails worked perfectly. Completely consistent on the affected types.
The bug report given to Claude
"Hovering a Shorts thumbnail on the home page does nothing. No toast. Regular thumbnails work fine. I inspected the element in DevTools and the tag is ytd-rich-grid-media."
✓ Root cause and fix
YouTube uses different custom element names across page types. The extension only handled a few. Claude expanded the recognised list to cover all known variants and added a fallback that searches any video container for a link — making the extension more resilient to future YouTube layout changes too.
💡 Use DevTools to give Claude the element name
"Doesn't work on some thumbnails" gives Claude nothing to work with. "The element tag is ytd-rich-grid-media" gives it exactly what it needs. Right-click → Inspect is the most useful move you can make when debugging UI issues.
◆ Phase 4 — Debug

Bug 3: Gemini Composer Click Landed Outside the Input

The PostMessage click was position-dependent. Smaller windows shifted where it landed.

🟥 The symptom
After Alt+Z, Gemini came into focus but the prompt wasn't pasted. The window was active — the click just didn't land in the text input. Happened consistently when the Gemini window was smaller than a certain size.
The bug report given to Claude
"Gemini gets focus but text isn't pasted. The window is active. This happens every time with a smaller window. When maximised, it works fine."
✓ Root cause and fix
Click coordinates were computed as fractions of window size, but the initial values were calibrated for one specific size. Claude extracted these into two named constants at the top of the file and added a hover event before the click to prime Chromium's input handling.
Result for anyone using this
If the click ever lands off-target on a different screen or layout, change two numbers at the top of the file. No understanding of the underlying implementation needed.
